Features · the nine modules

Everything an employee touches, in the order they touch it.

Cardinal HR is nine modules on one shared record — from the moment a role is posted to the day a cleared, papered, trained, enrolled, paid employee shows up in your directory, takes time off, and eventually offboards cleanly. Each module below lists what it does, what it can do, and the exact status words its records move through.

The nine modules

01Recruiting

Requisitions & applicant tracking

Open a requisition for the role you're filling, then move each applicant through a pipeline that won't let anyone skip a step or be quietly forgotten. Two state machines run side by side — one for the job, one for each person applying to it — and hiring a candidate is what fills the requisition and hands them off to screening.

  • Requisitions with headcount, department, and hiring manager
  • Enforced stages — no jumping straight to an offer
  • Reject or withdraw with a reason, kept on the record
  • A hire fills the role and starts a background check

Requisition status

open→ on_hold→ filled closed

Application status

applied→ screening→ interview→ offer→ hired
off-ramp rejected withdrawn
02Screening · HireCheck

FCRA-aware background checks

A background check is a consumer report under the FCRA, so HireCheck captures the candidate's disclosure and authorization before a single check runs. The data provider is a pluggable seam — Equifax is wired in today, and another provider drops into the same interface — and every result is retained with an audit trail.

  • Consent captured before any check is ordered
  • Nine check types, including drug screening, in packages
  • Pluggable provider — Equifax integrated
  • Pre-adverse notice, then a dispute window before adverse

Check status

authorized→ in_progress→ clear

Adverse path. A result that isn't clear doesn't auto-reject. It issues a pre-adverse notice and holds through a dispute window so the candidate can correct a wrong record before it costs them the job.

03MyHR · Onboarding

New-hire paperwork

MyHR assigns the forms a new hire actually has to complete and tracks each one from assigned to approved. The packet's status is derived from the forms inside it, the forms that require it collect a typed e-signature, and HR either approves a form or returns it with a note so the hire knows exactly what to fix.

  • I-9, W-4, and state tax withholding
  • Direct deposit and policy acknowledgments
  • Background-check consent collected here too
  • Typed e-signature on the forms that need one

Form status

assigned→ submitted→ approved
sent back returned

Returned isn't rejected. A returned form goes back to the new hire with HR's note and re-enters the flow at submitted once it's fixed — nothing is thrown away.

04Training

Course catalog & assessments

Build a customizable catalog of the courses you require — Sexual Harassment Prevention, safety, code of conduct — where each course is a set of lessons plus an optional passing assessment. Progress is tracked lesson by lesson per learner, and a course with an assessment isn't complete until the learner passes it.

  • Customizable catalog of required courses
  • Lessons plus an optional passing score
  • Lesson-by-lesson progress per learner
  • Completion you can prove, per employee

Enrollment status

not_started→ in_progress→ completed
assessment failed→ retake

Failing a required assessment holds the enrollment short of completed and opens a retake, so a completion always means the score was actually met.

05Benefits

Guided benefits election

Present a plan catalog with coverage tiers so a new hire elects or waives each plan in the same flow that hired and onboarded them. Dependents are added where a tier covers them, and the monthly premium is computed from the plan, tier, and dependents the employee actually chose.

  • Plan catalog with coverage tiers
  • Add dependents where the tier allows
  • Elect or waive each plan, on the record
  • Monthly premium computed from the elections

Enrollment status

not_started→ in_progress→ submitted→ confirmed

Waiving is a choice, not a gap. A declined plan is recorded as a waive so the enrollment is complete and the premium reflects exactly what was elected.

06Payroll

Gross-to-net calculation

Payroll computes each employee's gross-to-net from the same record — the reference configuration is Raleigh, NC, which applies federal income tax and North Carolina's flat rate with no local tax. It runs FICA, applies pre- and post-tax deductions to reach net pay, and calculates the employer-side taxes on top.

  • Federal + NC flat withholding (no local tax)
  • FICA — Social Security and Medicare
  • Pre- and post-tax deductions to net pay
  • Employer-side tax amounts computed too

Honest scope

Cardinal HR computes withholding — it does not file or remit taxes. The module produces the pay run and every tax and deduction figure, but sending money to and filing with tax authorities is handled by a registered payroll-tax provider connected to your account. Treat this as calculation software, not a tax service.

The Raleigh, NC setup is a worked reference configuration; other jurisdictions require their own tax rules before you run real pay there.

07Employee Directory · HRIS core

The shared employee record

The directory is the HRIS core every other module reads from and writes to — one record per person, so a candidate becomes an employee without anything being re-keyed. It holds departments and managers, derives the org chart from the reporting chain, and tracks employment status across the employee's time with you.

  • One employee record, referenced everywhere
  • Departments, managers, and contact details
  • Org chart derived from the reporting chain
  • Cycle-safe — no manager loops in the hierarchy

Employment status

active on_leave terminated

Cycle-safe by construction. The reporting chain is validated so no one can end up as their own manager, directly or through a loop — the org chart always resolves.

08Time Off · PTO

Leave & accrual balances

Define leave policies, accrue balances against them, and let employees request time off that a manager approves or denies. A balance is deducted when leave is approved and refunded if it's later cancelled or denied, so the number an employee sees is always what they can actually take.

  • Policies with accrual and per-employee balances
  • Requests approved or denied, never silently dropped
  • Balance deducted on approval, refunded on cancel
  • Overlap and insufficient-balance requests are refused

Request status

pending→ approved
off-ramp denied cancelled

The balance is the source of truth. Deductions and refunds are tied to the request's state, so a cancelled or denied day always returns to the balance — no manual reconciliation.

09Offboarding

Clean separations

The mirror of onboarding: open a separation case — voluntary, layoff, retirement — and work a checklist that makes sure nothing is missed on the way out. Each task is done or marked not-applicable, and the case's status is derived from the tasks rather than set by hand.

  • Return equipment, revoke access, collect the badge
  • Final paycheck, COBRA, benefits termination
  • Exit interview and knowledge transfer
  • Remove-from-directory closes the lifecycle

Case status

open→ in_progress→ completed

Derived, not asserted. A case is only complete when every task is done or explicitly N/A — so "offboarded" means the checklist was actually finished, not just clicked.

The shared thread

One application-to-alumnus record, an audit trail on every action.

The nine modules aren't nine products stitched together — they're nine views of the same person's file. Hiring provisions that one record into every module at once and remembers the id each assigns, so it resolves everywhere; and every state change above, from applied to offboarded, is appended to an immutable history. When someone asks "what happened," there's always an answer.
